# 产品案例：从海量聊天记录到可交付的 AI 关系产品

## 1. 项目摘要

拾光是一款本地优先的 AI 关系回顾应用。它把长期聊天整理为可搜索的记忆、可追溯的回答、关系时间线和可保存的深度报告。

我的角色不是单纯“提出需求”，也不是把 AI 生成的代码拼在一起。我负责从问题定义到发布验收的完整产品闭环：

- 识别真实用户任务与隐私约束；
- 设计普通搜索、AI 对话和 Deep Research 的分工；
- 决定数据模型、关系档案和摘要层级；
- 用真实失败现象驱动交互与工程修正；
- 建立测试、打包、版本和发布记录；
- 使用 AI 编程工具实现并验证产品。

## 2. 问题：记录很多，但很难形成理解

聊天软件擅长按时间存储，却不擅长跨时间理解。关键词搜索能够找到一条消息，但无法稳定回答：

- 某个主题如何从一次聊天变成长期共同关注？
- 什么时候关系出现明显变化？
- 一个结论由哪些原文支持？
- 如何把半年或一年的互动整理成结构化报告？

直接把十万级消息全部发送给模型既昂贵，也超出上下文，而且会放大隐私风险。因此问题不是“给聊天接一个大模型”，而是建立一套可靠的关系记忆系统。

## 3. 用户与 JTBD

### 核心用户

- 想回顾伴侣、朋友、家人或同学关系的人；
- 拥有大量长期聊天，但不会写脚本或操作数据库的人；
- 愿意使用 AI，但关心原文证据、数据隔离和隐私边界的人。

### Job to be Done

> 当我想理解一段重要关系时，我希望用自然语言提问，并随时回到真实日期和原文；当问题很大时，我希望得到覆盖全时段、可保存的完整报告，而不是一段没有证据的印象式回答。

## 4. 从 MVP 到产品的演进

| 阶段 | 最初做法 | 发现 | 产品转向 |
|---|---|---|---|
| 价值验证 | 静态统计与图表 | 有趣但只能回答预设问题 | 转向自然语言提问 |
| 规模处理 | 临时读取大量原文 | 上下文和成本不可控 | 建立日/月/年摘要金字塔 |
| 长问题 | 在对话框里输出长文 | 无任务状态、不能保存 | 独立 Deep Research 与报告库 |
| 分发 | Python 脚本 | 非技术用户无法操作 | 本地网页应用 + 单文件 EXE |
| 多用户 | 直接复制情侣版本 | 文案误读普通关系、数据可能串档 | 通用/情侣两种语义，每联系人独立存储 |
| 真实导入 | 只要加一个按钮 | 版本、路径、权限、耗时和失败都暴露给用户 | 首次向导、真实进度、错误保留与兜底 |
| 发布 | 每改一次重新打包 | 副本混乱、无法确认版本 | 开发快循环与发布慢循环分离 |

## 5. AI 不是一个按钮，而是一套任务路由

我把问题分成五类：

1. **不需要历史的普通交流**：直接回答，禁止假装查过数据库。
2. **模糊事件与变化**：先搜摘要，缩小日期范围。
3. **真实原话与证据**：查原文，返回日期、发送者和上下文。
4. **数量与分布**：执行受限的只读 SQL，避免模型心算。
5. **全时段研究**：进入后台 Deep Research，逐月扫描、定位关键日、回到原文取证、生成 PDF。

这套路由同时解决三件事：成本、可靠性和用户预期。用户知道普通对话为什么快，Deep Research 为什么慢，也知道结论能否回到原文。

## 6. 摘要金字塔：把技术限制转成产品能力

日摘要保存当天的事件、话题、双方情绪线索、代表性原话和标签；月摘要聚合趋势与关键日期；年摘要形成主题索引。

它不是为了“压缩得更短”，而是为三个产品场景服务：

- 快速定位某件事发生在哪段时间；
- 观察跨月变化而不丢失原文入口；
- 让 Deep Research 覆盖全时段，再对重点日期下钻。

## 7. 最有价值的几次问题修正

### “导入卡退”并不是导入太慢

最初看到选联系人后页面退回，我也怀疑是十万级消息导致性能问题。实测发现大联系人读取只需数秒；真正失败的是“通讯录有此人，但消息分片没有对应表”。后端很快返回错误，而前端两秒后自动跳页，用户才感觉闪退。

最终方案：准备阶段先扫描消息表，只显示能够导入的人；导入变成可轮询后台任务；页面显示真实阶段、百分比和条数；错误不消失，提供重试与返回。

产品反思：不要用用户看到的现象直接命名根因。“慢”和“快速失败但没解释”体验相似，解法完全不同。

### 更新客户端后自动扫描失败

固定猜测常见目录在开发机上有效，却漏掉了从旧版迁移并把数据放在其他磁盘的用户。修复不是再增加几个猜测路径，而是读取上游应用自己的配置文件，再用常见路径兜底。

产品反思：自动化的第一原则是读取真实状态，不是把猜测写得越来越长。

### 双击新版却仍打开旧版

文件、进程和用户数据是三件不同的事。新 EXE 不会替换已经加载到内存的旧进程；启动器又只比较“通用版/情侣版”，没有比较具体构建。真实 Windows 验收还发现默认 HTTP 服务允许地址复用，两个后端可能共享端口。

最终方案：打包写入构建编号；只有相同构建才复用；本地 HTTP 服务使用独占端口，新版遇到旧版时自动切换备用端口。

产品反思：版本身份必须进入运行时，而不是只写在文件名里。

## 8. 隐私与伦理边界

- 默认本地存储聊天库、AI 会话、报告和设置；
- 每个联系人完全隔离；
- API Key 不进入浏览器，Windows 生产版使用当前用户绑定的加密存储；
- 对外模型请求只发送当前任务必要上下文；
- 用户可以排除日期和关键词范围；
- 不注入无法审计的第三方二进制，即使这意味着主动放弃照片墙；
- 公开作品仓库不包含真实聊天、密钥、报告和原生密钥恢复实现。

## 9. 结果与证据

- 私有生产环境验证过十万级消息和数百联系人规模；
- 形成今天页、原文搜索、摘要搜索、时间线、仪表盘、普通 AI 对话、Deep Research、报告库和关系提醒；
- 普通 AI 对话、置顶记忆、用户纠正、未来信和报告任务持久化；
- 生产工程形成 63 项自动化测试；
- 通用关系版与情侣版完成 Windows 单文件分发；
- 建立变更记录、产品决策记录、发布清单和唯一发布入口。

## 10. 我展示的不是功能数量

这个项目最能代表我的地方，是我逐渐把“做出东西”转化成“构建可解释、可失败、可恢复、可发布的产品”：

- 从真实用户现象找根因，而不是立即加功能；
- 把模型能力、数据库能力和交互反馈设计成一条链；
- 主动记录错误、边界和不做的事；
- 用工程验证保护产品决策；
- 在 AI 协作开发中保持产品负责人对目标和风险的控制。
