# 产品决策记录

## PD-001：终点是关系洞察，不是聊天导出

- **背景**：导出和统计容易展示，却不能回答临时问题。
- **决策**：产品围绕“提问—取证—回顾—行动”组织，导入只是前置能力。
- **结果**：功能优先级从图表数量转向搜索、证据、时间线和报告。

## PD-002：采用日—月—年摘要金字塔

- **背景**：十万级消息不能直接塞进模型上下文。
- **备选**：只做向量检索；每次临时总结；固定摘要层。
- **决策**：日摘要保留细节，月摘要保留变化，年摘要作为主题索引。
- **代价**：首次分析更慢，需要增量更新和失败恢复。

## PD-003：本地网页应用优先，不先做 Electron

- **背景**：核心价值尚在验证，换壳会增加构建和调试成本。
- **决策**：Python 本地服务 + 浏览器 UI，打包时再做单文件应用。
- **结果**：业务层跨平台，Windows 快速交付；代价是必须设计浏览器页签与后端生命周期。

## PD-004：每个联系人必须独立存储

- **背景**：不同关系不能共享聊天、AI 会话、报告或 Key。
- **决策**：注册表只存元数据，每个档案拥有独立数据根。
- **结果**：档案切换通过重启当前本地实例生效，避免模块误读旧路径。

## PD-005：不同问题使用不同检索方式

- **背景**：单一向量搜索无法同时保证原文准确、统计可靠和全时段覆盖。
- **决策**：原文检索、摘要定位、只读 SQL 和 Deep Research 并存，由任务类型路由。
- **结果**：成本与可信度更可解释，回答可回到证据。

## PD-006：真实进度，不使用随机假进度

- **背景**：导入和报告可能持续数秒到数分钟。
- **决策**：后台任务上报真实阶段、比例和已处理条数；错误必须留在当前页面。
- **结果**：等待变得可理解，失败可以重试。

## PD-007：不注入未审计二进制

- **背景**：照片解密曾需要注入来源不明的第三方组件。
- **决策**：主动放弃照片墙，不以功能完整换取不可控安全风险。
- **结果**：产品少一个功能，但隐私承诺更可信。

## PD-008：公开作品集与私有生产工程分离

- **背景**：原工程含真实聊天、数据库、报告、密钥配置和高风险数据适配器。
- **决策**：用全新 Git 历史创建公开版，只保留合成数据、产品层和安全源码。
- **结果**：面试官能运行和审阅产品，同时不泄露用户或传播未经授权的数据访问能力。

## PD-009：开发快循环与发布慢循环分离

- **背景**：每次修改都打包，导致多份 EXE 无法确认新旧。
- **决策**：日常只跑源码和测试；一批需求验收后再统一打包、校验、归档。
- **结果**：减少重复时间，版本与校验值可追溯。

## PD-010：AI 协作必须留下产品记录

- **背景**：快速迭代容易只剩代码，看不到为什么做和如何判断。
- **决策**：每次迭代记录用户现象、假设、证据、修正和反思。
- **结果**：项目成为可以复盘的产品作品，而不只是一次性应用。
