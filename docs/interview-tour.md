# Interview Tour：如何在 5 分钟介绍拾光

## 0:00–0:40｜一句话与用户问题

“拾光是一个本地优先的 AI 关系回顾产品。它解决的不是聊天导出，而是长期聊天有记录、没理解的问题：用户可以问关系变化、找到真实原话，或生成覆盖全时段的 PDF 报告。”

## 0:40–1:30｜展示产品

先展示 README 的“今天页”和“仪表盘”：

- 今天页把摘要、关系天气和下一步行动放在一起；
- 仪表盘强调关系节奏，而不是堆图表；
- 聊天记录支持原文、整句、条件、摘要和 AI 检索；
- Deep Research 是独立长任务，不挤在普通对话里。

## 1:30–2:30｜讲一个核心产品判断

“我没有把十万条消息全部塞给模型，而是设计日—月—年摘要金字塔。模糊问题先从摘要定位，需要证据时回到原文；数量问题用只读 SQL；完整研究走独立 Deep Research。这让成本、速度和可信度都可解释。”

## 2:30–3:30｜讲一个错误与修正

“用户说导入卡退时，我一开始以为是消息多。实测发现大联系人读取很快，真正问题是某些通讯录联系人没有消息表，后端快速失败后页面又自动跳走。我最后做的是提前过滤、真实进度、错误保留和重试，不是加一个假的加载动画。”

## 3:30–4:20｜隐私与边界

“关系数据高度敏感，所以每个联系人独立存储，本地服务只监听 loopback，Key 不进浏览器，用户还能排除日期或关键词。我还主动放弃了依赖未审计二进制的照片功能。公开仓库使用全新 Git 历史和合成数据。”

## 4:20–5:00｜工程化与成长

“项目早期我一直改完就打包，版本很乱。后来我把开发快循环和发布慢循环分开，用测试、构建号、SHA256 和发布记录保护产品决策。这个项目最能体现的是，我能在 AI 协作开发中持续定义问题、验证假设、管理风险并把原型推到可交付状态。”

## 面试官可能追问

### 为什么不用向量数据库解决所有问题？

因为原文、统计、长期变化和完整研究对准确性与覆盖率的要求不同。单一检索方式会让产品成本和错误不可解释。

### 你如何评估 AI 回答？

当前以证据可追溯、日期覆盖、只读统计正确性和用户纠正是否进入后续上下文为核心；下一步会建立固定问题集和事实一致性指标。

### 为什么做本地网页应用？

它能用最小工程成本验证产品价值，并保留跨平台业务层；代价是必须补浏览器生命周期、本地 HTTP 安全和端口管理。

### 公开版为什么没有原生导入实现？

这是有意识的发布边界：公开作品集要展示产品与架构，同时避免真实私人数据和高风险适配器进入公共历史。
