# 研发与产品复盘日志

这份日志故意保留“走错的路”。它不是版本更新列表，而是我如何从用户现象形成产品判断的证据。

## 阶段 1：从真实数据验证价值

- **做法**：先完成本地数据清洗、统计与静态报告。
- **收获**：真实规模能产生有趣洞察，但固定图表无法回答临时问题。
- **转向**：把核心入口改成自然语言提问与可搜索原文。

## 阶段 2：摘要金字塔

- **问题**：完整聊天超过上下文窗口，临时总结重复且昂贵。
- **方案**：预计算日摘要，聚合为月摘要与年摘要。
- **反思**：预计算不是纯性能优化，它决定用户能否跨时间理解关系。

## 阶段 3：Deep Research 与 PDF

- **问题**：普通对话适合短问题，不适合十分钟级全库研究。
- **方案**：独立任务队列、进度、失败、重试、取消、报告库与 PDF。
- **反思**：长任务必须成为产品对象，而不是一次超长 HTTP 请求。

## 阶段 4：从脚本到非技术用户产品

- **问题**：Python 脚本对朋友不可用。
- **方案**：首次向导、本地网页 UI、单文件 EXE、数据目录自动初始化。
- **反思**：少暴露文件不等于产品化；权限、等待、失败和版本兼容都要有交互。

## Bug 复盘：用户说“导入卡退”

- **错误假设**：消息太多，导入性能不足。
- **证据**：大联系人读取正常；无消息表联系人会在几十毫秒内失败，页面随后自动跳走。
- **修正**：提前过滤、后台任务、真实进度、错误保留、重试与返回。
- **反思**：先测时间线，再决定优化性能还是失败体验。

## Bug 复盘：主页打开但没有任何按钮响应

- **根因**：恢复代码时重复声明前端变量，JavaScript 在解析阶段中止。
- **修正**：删除重复块，为两个 HTML 的内联脚本加入 `node --check` 回归。
- **反思**：“HTTP 200”不等于产品可用，前端语法和真实交互必须进验收。

## Bug 复盘：短窗口侧栏无法滚动

- **根因**：侧栏允许内容溢出，页面根节点又禁止滚动。
- **修正**：侧栏独立滚动并做真实窗口尺寸验收。
- **反思**：设计稿上的完整页面不能代表用户真实窗口。

## Bug 复盘：推荐问题遮挡对话

- **根因**：推荐词多行换行且不属于回答滚动区。
- **修正**：改为单行横向、可收起；纵向滚轮转交给对话区。
- **反思**：推荐是辅助，不应抢占主任务空间。

## Bug 复盘：升级客户端后自动找不到数据

- **错误修正**：最初只增强手动选择和错误文案。
- **真正根因**：固定猜测目录没有读取上游应用记录的自定义存储路径。
- **最终修正**：优先读取真实配置，再用常见路径兜底。
- **反思**：改善失败体验不能冒充自动化已经修好。

## Bug 复盘：新版 EXE 仍打开旧界面

- **根因 1**：启动器只比较产品类型，没有构建编号。
- **根因 2**：Windows 上默认 HTTP 服务允许地址复用，两个后端可能共享端口。
- **修正**：构建身份、同构建复用、异构建备用端口、独占端口绑定。
- **反思**：文件版本、运行进程和用户数据必须分别管理。

## 流程复盘：为什么一直“改—打包—再改”

- **根因**：没有区分开发快循环与发布慢循环，也没有唯一发布命令。
- **修正**：源码验证 → 自动测试 → 一批需求验收 → 构建 → 多目录同步 → SHA256 → 发布记录。
- **反思**：工程化不是增加文档，而是让正确动作更容易重复。

## 公开发布复盘

- **风险**：直接公开原 Git 历史，即使删除当前数据库，也可能在旧提交泄露真实聊天与 Key。
- **决策**：建立全新公开仓库和全新历史；生成合成数据；敏感适配器只保留接口。
- **反思**：公开源代码也是一次产品发布，需要用户、边界、风险和验收标准。
